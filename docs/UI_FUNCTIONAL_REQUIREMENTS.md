# 7G Tool Platform 用户功能说明

本文档只描述用户可见、可操作的功能，不描述代码实现、接口结构、目录结构或技术方案。目标是为 UI 设计提供直接依据。

## 1. 产品定位

这是一个“工具统一运行平台”。

用户进入平台后，可以在同一个界面中：

1. 查看可用工具
2. 按分类筛选工具
3. 搜索工具
4. 查看工具说明
5. 填写参数并运行工具
6. 观察任务执行状态
7. 查看执行结果
8. 查看和下载产出文件
9. 导入新的插件工具
10. 配置当前使用环境

## 2. 目标用户

主要面向以下用户：

1. 需要频繁运行脚本工具的内部业务人员
2. 需要处理标注、清洗、转换、统计、导出任务的操作人员
3. 需要把多个脚本统一管理的团队成员

这些用户的典型特点是：

1. 不希望记命令行
2. 希望通过表单运行工具
3. 希望看到任务是否成功
4. 希望直接拿到结果文件

## 3. 用户的核心任务

从用户角度，平台主要支持 5 类任务：

1. 找到需要的工具
2. 理解这个工具怎么用
3. 提交一次运行
4. 查看执行结果和产物
5. 新增一个可用工具

## 4. 页面总体结构

平台可以理解为一个单页工作台，主要由以下区域组成：

### 4.1 顶部区域

顶部区域用于承载平台级操作：

1. 平台名称与品牌标识
2. 当前环境显示
3. 环境配置入口
4. 插件导入入口

### 4.2 左侧区域

左侧区域用于工具分类导航：

1. 展示工具分组列表
2. 支持点击切换当前分组
3. 支持高亮当前选中的分组
4. 需要包含“全部”分组

### 4.3 中间主区域

主区域用于展示工具卡片和筛选能力：

1. 搜索框
2. 状态筛选
3. 工具列表卡片

### 4.4 右侧区域

右侧区域用于展示执行过程信息：

1. 运行中的任务队列
2. 最近执行记录

## 5. 工具浏览功能

### 5.1 查看工具列表

用户进入页面后，应能看到工具列表。

每个工具至少应展示：

1. 工具名称
2. 简短描述
3. 工具状态
4. 版本信息
5. 操作按钮

### 5.2 工具分组浏览

用户可以通过左侧分组列表快速浏览工具。

交互要求：

1. 点击某个分组后，只显示该分组下的工具
2. 点击“全部”后，显示所有工具
3. 当前分组有明确选中态

### 5.3 工具搜索

用户可以通过搜索框查找工具。

搜索应尽量覆盖：

1. 工具名称
2. 工具描述
3. 工具标签

### 5.4 工具状态筛选

用户可以按工具状态筛选。

建议支持的状态概念：

1. 全部
2. 可用
3. 草稿
4. 停用

## 6. 工具卡片功能

每个工具卡片至少需要两个动作：

1. 查看使用说明
2. 运行工具

### 6.1 查看使用说明

用户点击“使用说明”后，应打开说明弹窗或详情面板。

说明内容至少包括：

1. 工具名称
2. 工具用途
3. 参数说明
4. 使用提示
5. 工具分组
6. 工具状态
7. 运行方式或类型

设计目标：

1. 让用户在不离开当前页面的前提下理解工具怎么用
2. 让首次使用者能快速完成一次运行

### 6.2 运行工具

用户点击“运行工具”后，应打开运行弹窗或运行面板。

这个区域的目标不是展示技术细节，而是让用户完成一次参数填写和提交。

## 7. 运行工具功能

### 7.1 打开运行表单

用户进入运行界面后，应看到一个动态表单。

表单字段来自工具定义，但从用户视角需要支持以下体验：

1. 字段有中文标签或可理解名称
2. 必填项有明显标识
3. 字段有默认值时自动带出
4. 不同类型字段用不同控件展示

### 7.2 支持的输入类型

从用户视角，平台至少要能支持以下输入方式：

1. 单行文本
2. 多行文本
3. 密码输入
4. 下拉选择
5. 布尔开关或真假选择
6. 路径输入
7. 多文件或多路径输入

即使底层实现不同，UI 设计时应把它们视为标准表单控件。

### 7.3 参数填写体验

运行表单建议满足以下交互要求：

1. 用户能看懂每个字段的含义
2. 用户知道哪些字段必须填
3. 用户能取消本次操作
4. 用户能提交本次运行

### 7.4 提交运行

用户点击“开始执行”后，系统应立即给出反馈。

理想反馈包括：

1. 已成功提交
2. 任务进入等待状态
3. 后续可在运行队列中看到该任务

如果提交失败，应明确提示：

1. 失败原因
2. 是参数问题还是系统问题

## 8. 执行状态与任务反馈

### 8.1 运行队列

页面右侧应有“运行队列”区域，用来展示当前正在进行的任务。

每条任务至少展示：

1. 工具名称
2. 当前状态

典型状态包括：

1. 等待执行
2. 运行中
3. 成功
4. 失败
5. 超时

### 8.2 最近执行记录

页面右侧还应有“最近执行”区域。

它的作用是帮助用户快速回看最近发生了什么。

每条记录建议展示：

1. 工具名称
2. 结果状态
3. 任务标识
4. 日志信息或简要说明

### 8.3 长任务感知

对于执行时间较长的任务，用户需要明确知道：

1. 任务已经发出
2. 任务还在执行
3. 不需要重复点击提交

因此 UI 应设计出明显的状态流转，而不是只在最后给结果。

## 9. 执行结果查看

### 9.1 结果弹窗

任务结束后，应弹出结果窗口或结果面板。

结果区域至少需要显示：

1. 工具名称
2. 执行状态
3. 标准输出内容
4. 错误输出内容
5. 任务 ID
6. 日志文件信息
7. 退出码

### 9.2 标准输出

如果任务执行成功，用户通常最关心工具输出了什么。

因此需要：

1. 大块可读文本区域
2. 支持滚动查看
3. 支持复制输出内容

### 9.3 错误输出

如果任务失败或超时，用户需要更明显地看到错误信息。

错误区域需要：

1. 与普通输出有视觉区分
2. 突出失败原因
3. 保留完整错误文本

### 9.4 复制结果

用户应能一键复制输出内容。

适用场景：

1. 发给开发排查
2. 复制到群里
3. 复制到文档中留痕

## 10. 产出文件查看与下载

很多工具的核心价值在于“生成文件”。

因此结果区域必须考虑产物展示。

### 10.1 产物列表

如果任务有生成文件，应显示产物列表。

每个产物建议展示：

1. 文件名
2. 文件大小
3. 文件类型

### 10.2 产物操作

每个产物至少支持两个操作：

1. 在线查看
2. 下载文件

### 10.3 产物排序

从用户体验看，产物可以按“更容易直接阅读的文件类型”优先展示，例如：

1. HTML
2. PDF
3. 文档
4. Markdown
5. JSON
6. 其他文件

这样用户更容易先看到最终结果，而不是原始中间文件。

## 11. 插件导入功能

平台除了运行已有工具，还支持导入新工具。

这是一个明确的用户功能，不应只当成后台能力。

### 11.1 导入入口

用户可以点击“导入插件”按钮进入导入流程。

### 11.2 导入方式

导入方式需要至少支持 3 类：

1. 本地目录
2. ZIP 文件
3. Git 仓库

### 11.3 导入表单

用户在导入界面需要完成以下选择和输入：

1. 选择导入方式
2. 输入对应路径或仓库地址
3. 可选填写分支或 Tag
4. 选择是否允许覆盖同名插件
5. 选择是否自动安装依赖

### 11.4 导入结果反馈

导入成功后，用户希望立即知道：

1. 哪个插件导入成功
2. 是否完成依赖安装
3. 新工具是否已出现在工具列表中

导入失败后，用户希望看到：

1. 失败原因
2. 是路径问题、格式问题，还是安装失败

## 12. 环境配置功能

平台存在“环境配置”概念，用于切换或记录当前使用环境。

从用户角度，这个功能用于管理不同执行环境下的连接信息。

### 12.1 环境入口

用户可点击顶部“环境”按钮打开环境配置。

### 12.2 环境字段

当前功能语义上包含两类配置：

1. 标注平台配置
2. SSH 连接配置

用户需要填写的内容通常包括：

1. 环境名称
2. 标注平台地址
3. 标注平台账号
4. 标注平台密码
5. SSH 地址
6. SSH 用户名
7. SSH 密码

### 12.3 环境操作目标

这个功能对用户的意义是：

1. 保存当前常用环境信息
2. 避免每次手工重复输入
3. 在不同环境之间切换

### 12.4 设计建议

虽然这里只描述功能，不限定实现，但 UI 上建议把环境配置做得更像“可管理资源”，而不是一次性的临时表单。

## 13. 用户完整操作路径

下面给出最核心的用户路径，UI 设计时应优先保证这些路径顺畅。

### 13.1 路径一：查找并运行一个工具

1. 用户进入平台首页
2. 按分类或搜索找到目标工具
3. 查看工具说明
4. 点击运行工具
5. 填写参数
6. 提交任务
7. 在右侧看到任务进入队列
8. 等待任务结束
9. 查看结果
10. 下载或查看生成文件

### 13.2 路径二：修正一次失败的运行

1. 用户提交工具运行
2. 任务失败
3. 用户在结果窗口看到错误输出
4. 用户关闭结果窗口
5. 重新打开运行表单
6. 调整参数
7. 再次提交

### 13.3 路径三：导入一个新插件

1. 用户点击导入插件
2. 选择导入方式
3. 填写路径或仓库地址
4. 设置覆盖和安装选项
5. 提交导入
6. 查看导入成功或失败信息
7. 返回工具列表，确认新工具出现

### 13.4 路径四：配置当前环境

1. 用户点击环境按钮
2. 打开环境配置弹窗
3. 修改或填写连接信息
4. 保存配置
5. 回到主界面继续使用

## 14. 状态设计建议

为了让 UI 更完整，建议从设计层面明确以下状态。

### 14.1 工具状态

1. 可用
2. 草稿
3. 停用

### 14.2 任务状态

1. 等待执行
2. 运行中
3. 成功
4. 失败
5. 超时

### 14.3 页面状态

1. 工具列表加载中
2. 工具列表为空
3. 搜索无结果
4. 导入成功
5. 导入失败
6. 提交成功
7. 提交失败
8. 结果可查看

## 15. 设计时特别要照顾的体验点

### 15.1 降低命令行感

虽然平台底层是在运行脚本，但用户界面不应让人感觉自己在“填命令行参数”。

UI 应尽量让它看起来像：

1. 选择一个工具
2. 填一个业务表单
3. 得到一个业务结果

### 15.2 强化结果导向

用户最终最关心的通常不是执行过程本身，而是：

1. 是否成功
2. 输出了什么
3. 文件在哪里

因此结果展示和产物下载应比技术细节更突出。

### 15.3 对失败场景友好

很多工具用户不是开发者。

失败时应帮助用户：

1. 看懂失败
2. 知道下一步怎么办
3. 方便复制信息给他人排查

### 15.4 支持高频重复操作

该平台很可能存在大量重复执行场景，因此 UI 设计可以考虑：

1. 保留最近使用痕迹
2. 突出最近执行记录
3. 让再次运行动作更顺手

## 16. 给 UI 设计师的最终摘要

如果把这个产品抽象成一句话：

这是一个“面向内部使用者的工具运行工作台”，核心体验不是浏览内容，而是“快速找到工具、填写参数、发起运行、拿到结果文件”。

如果把设计重点再压缩成 4 个关键词，就是：

1. 找工具
2. 跑工具
3. 看结果
4. 拿产物

如果后续 UI 设计要分优先级，建议优先保证这四条主链路：

1. 工具发现链路
2. 运行提交链路
3. 状态反馈链路
4. 结果与产物链路

# 项目全局知识库 (Project Global Knowledge Base)

本文档整理并记录了目前已知的所有关于当前 AI 目标检测项目的信息。它涵盖了项目的业务背景、技术栈、核心工作流以及近期的架构演进历史。

## 1. 项目背景与业务目标

本项目是一个基于计算机视觉（CV）的目标检测工程，核心业务聚焦于**城市管理（城管）**领域的自动化监控与识别。主要涉及两大检测场景：
- **街道垃圾检测**：包括散落垃圾 (`garbage_street_scattered`)、垃圾桶 (`garbage_street_can`)、垃圾堆 (`garbage_street_dump`)、袋装垃圾 (`garbage_street_pack`)、纸箱 (`object_box`) 以及非垃圾目标 (`non_garbage`)。
- **占道经营检测**：识别摊贩占道经营行为（如 `streetVendor_catering` 等）。

**终极目标**：打通从“前线数据采集”到“边缘设备部署”的全链路，实现高效迭代，最终将量化后的模型（如 ONNX 格式）稳定部署在算力受限的端侧设备（如 BM1688 边缘计算盒）上。

## 2. 核心技术栈

- **算法框架**：Ultralytics YOLOv11（目前主要使用轻量级的 YOLOv11s 版本）。
- **标注平台**：LabelU（作为核心的人工复核与标注平台，输出 JSON 格式）。
- **预标注辅助**：结合了旧版 YOLO 推理以及前沿的 VLM（视觉大语言模型）进行机器预标注。
- **模型量化与部署**：ONNX Runtime，支持 CPU/GPU 及边缘设备的异构部署测试。
- **脚本语言**：全系使用 Python 编写，涉及 `xml.etree` (处理 VOC XML), `PIL` (图像处理), `requests` (调用 LabelU API) 等。

## 3. 标准化数据流与文件结构

经过近期的深度重构，项目告别了过去杂乱的“脚本全家桶”时代，建立了一套严格的 **8 阶段标准化流水线**（位于 `模型训练标准化流水线/` 目录下）：

| 阶段编号 | 阶段名称 | 核心职责 | 代表性工具库 |
| :--- | :--- | :--- | :--- |
| **01** | **数据采集与切片清洗** | 视频抽帧、去重(`dedup.py`)、清洗问题图、统一分辨率 | `mp4cimg.py`, `resize_images...` |
| **02** | **数据预标注** | 机器全自动预标图库，并**从平台下载资源** | `labelu_download_images/`, `vlm_preannotate/` |
| **03** | **数据标注辅助** | LabelU 平台高级管理（删废片、查询进度、统一下发工具） | `labelu_undo_count/`, `labelu_recttool_unify/` |
| **04** | **数据质检与分析** | 格式互转（JSON转XML）、排查丢框/错标、校验标签配置 | `01_convert_annotations.py`, `labelu_tagtool_check/` |
| **05** | **数据预处理** | 融合多位标注员的数据、合并重命名、划分 7:2:1 的 YOLO 数据集 | `01_merge_team_data.py`, `prepare_yolo_dataset.py` |
| **06** | **模型训练** | 读取 YAML 启动模型训练 | `train.py`, `train.yaml` |
| **07** | **模型评估** | 云端全面 mAP 评估、搜索最佳阈值、挖掘 False Positives/Negatives | `cloud_based_evaluation.py`, `failure_mining_yolo.py` |
| **08** | **模型量化与部署** | 导出 ONNX 模型，并执行端侧设备的纯文本模拟评估 | `onnx_transform.py`, `pythonTest_evaluation.py` |

> [!NOTE]
> 原先混合多阶段职责的聚合脚本（如 `run_preprocess.py`）已被彻底剥离废弃，强制推行单一职责原则。

## 4. 重要的业务难点与解决方案（近期历史）

基于过往的迭代记录，项目在推进过程中攻克了以下关键问题：

1. **海量数据的均衡分配：** 编写了 `distribute_images.py` 等脚本，将海量原始切片按序、平均地下发给多名标注员（如划分至 `cc2`, `lxl2`, `sy2` 等 8 个目录）进行并行处理。
2. **标注平台规范失控：** LabelU 在多人协作时容易出现类别大小写拼写错误（如 `StreetVendor` 和 `streetvendor`）、缺失 `<size>` 或标签工具配置不一致。为此开发了专属的 **质检与对齐高级工具包** (`labelu_tagtool_unify`, `fix_problem_json.py`)。
3. **模型评估漏类Bug (Missing Class)：** 曾遇到在评估环节某类别（如 `object_cabinet`）丢失统计的问题，通过完善底层解析与 XML 配对脚本予以了修复。
4. **OOM (内存溢出) 调优：** 在双 GPU 服务器训练 YOLOv11 时，经历过僵尸进程和 OOM 报错，通过优化 Batch Size 和多进程释放策略稳定了训练流。

## 5. 待办与未来拓展预留

- **数据增强模块 (Data Augmentation)：** 在现有的 8 阶段流水线中，仍预留有拓展离线数据增强的空间。
- **自动化循环迭代 (Data Engine)：** 目前 07 阶段发现的坏案 (Failure Mining) 还需要人工手动导回 02 阶段，未来可以通过脚本实现完全闭环的自学习数据飞轮。
